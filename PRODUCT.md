# RADAR 产品说明

## 产品定位

RADAR 是一套面向金融研发交付的**投产版本全生命周期管控平台**。它解决工作项分散、阶段状态不可见、评审材料不可追溯以及跨机构协同依赖人工催办的问题。

产品不替代代码仓库、测试平台或 PAMS 等专业系统；它负责把这些系统和人工填写的交付信息收敛为以投产点为中心、可以追踪和审计的协作记录。

## 用户与使用情境

主要用户是云南农信、建信金科及参与投产管理的业务、开发、测试、运维、项目和评审人员。用户以桌面端高频处理列表和表单，平板、手机用于查看和轻量操作。

系统支持一人多角色。默认角色覆盖业务、开发、测试、运维、机构负责人、需求/架构/测试/项目/安全/质量管理、配置管理员、管理员与超级管理员；实际可见菜单和可执行操作由角色权限矩阵决定。

## 领域对象与关系

| 对象 | 作用 | 与其他对象的关系 |
| --- | --- | --- |
| 投产点 | 版本窗口与全局筛选单位。 | 需求、工单和投产申请均关联投产点；审批以申请投产点定位。 |
| 需求 | 业务需求的分析入口，编号通常为 `RC_…`。 | 可拆分开发/测试任务，被投产申请引用。 |
| 工单 | 外部问题或工单口径的分析入口，编号通常为 `TK_…`。 | 与需求共享开发、测试、投产链路，但保持独立字段与编号。 |
| 开发任务 | 按实施系统承接工作项的开发工作。 | `req_code` 指向需求或工单编号；支撑影响性分析与测试。 |
| 测试任务 | 按 SIT、UAT、NFT、SEC 管理测试。 | `req_code` 指向需求或工单编号；NFT、SEC 按需创建。 |
| PAMS 问题 | 外部同步的问题快照。 | 可被投产申请引用，不强制经过开发、测试承接。 |
| 投产申请 | 变更单和制品的汇总视角。 | 关联需求、工单或问题，记录变更系统、交付制品、摆渡状态。 |
| 投产审批 | 可评审、可签署、可投产的对象视角。 | 由投产申请的关联对象展开；首开时创建审批任务和会签项。 |
| 附件、签名、审计日志 | 可证明交付过程的证据。 | 按实体和字段归档，贯穿各阶段。 |

## 业务流程

### 需求与工单

```text
登记 → 分析 → 分析完成 → 开发承接 → 开发完成 → 测试承接 → 测试完成 → 投产申请 → 审批会签 → 已投产
```

需求和工单的分析字段与编号规则不同，但自开发承接起共享后续链路。一个工作项可以按系统拆成多个开发或测试任务，因此“任务完成”与“工作项完成”必须按对应页面和状态规则理解，不能简单按单条记录推断。

### 测试

SIT、UAT 是常见阶段；NFT 与 SEC 是按需测试。测试任务可按系统拆分，也可以按整体承接。各阶段状态、负责人、计划/实际时间、附件和排期偏差率都应记录在任务上。

### 问题与投产

PAMS 问题可同步进 RADAR，作为问题快照浏览和引用。问题不必经过需求/工单的开发、测试链路，但可直接进入投产申请和审批。

投产申请记录变更单、关联对象、系统、制品、交付单位和摆渡信息。投产审批将每个关联对象作为独立审批单元，并按配置的会签角色创建会签项。评审结论会同步影响投产状态：评审同意可由待评审推进至待投产；投产登记形成最终闭环。

## 模块与用户结果

| 模块 | 用户能够完成的事 |
| --- | --- |
| 效能仪表盘 | 查看工作项、开发、测试、制品的指标；配置个人或系统图表，按维度聚合、筛选、透视并钻取明细。 |
| 版本概览 | 在一个投产点内跨需求、工单和问题查看链路状态、实施机构、详情和历史，并导出宽表。 |
| 需求分析 / 工单分析 | 建档、维护分析、关联系统与负责人、导入导出；被任务承接后保护关键编号和删除操作。 |
| 开发管理 | 按系统接收工作项、维护开发状态与排期、填写影响性分析、管理阶段附件。 |
| 测试管理 | 分别维护 SIT、UAT、NFT、SEC 的承接、状态、覆盖分析、附件与排期。 |
| 投产申请 | 汇总变更系统、制品、摆渡、关联需求/工单/问题和投产点。 |
| 投产审批 | 查看审批对象、会签、签名、投产状态与材料，导出评审单。 |
| 问题管理 | 同步、查询和清理 PAMS 问题快照。 |
| 人员管理 / 系统设置 | 管理用户、角色、权限、字典、系统、投产点、平台配置和动态必填项。 |

## 产品原则

1. **投产点优先**：默认筛选和统计围绕投产窗口，跨窗口结果必须明确展示口径。
2. **阶段可核对**：每个列表和详情都要让用户快速看到当前状态、负责人、系统和下一步。
3. **可追溯优先于便利的绕过**：关键变更、附件、签名和会签结论必须留痕；不能靠前端隐藏来替代规则校验。
4. **配置驱动而非硬编码**：流程状态、字典、角色权限、编号模板和必填项都是可配置业务规则；新增功能应尊重这一边界。
5. **为高频业务操作优化**：密集列表、筛选、导入导出、原位编辑和多页签工作区优先于展示性界面。

## 关键验收口径

- 从任一需求或工单可以追到其开发、测试、投产申请和审批信息；从投产审批也可以回看来源对象与材料。
- 任何写操作都在服务端执行权限、状态和必填项校验；关键业务字段的创建、修改、删除可在历史中核对。
- 版本概览和仪表盘对投产点筛选的结果一致，NFT/SEC 缺失只表示未按需承接。
- 导入、导出、详情和列表对系统、投产点、状态等字段使用相同的解析和展示口径。
- 系统图表由具备 `dashboard:manage` 权限的用户维护；个人图表仅对创建者可见。
