# RADAR 多人共建指南

> 适用对象：同时在同一代码库开发 RADAR 的工程师与开发助手。  
> 目标：让功能可以并行交付，同时保持流程口径、权限、安全和数据库双适配的一致性。

本文件是协作的约束性说明；产品语义见 [PRODUCT.md](PRODUCT.md)，视觉规则见 [DESIGN.md](DESIGN.md)，快速开发入口见 [AI-GUIDE.md](AI-GUIDE.md)。

## 1. 先统一的工程事实

- 前端是 `web/` 下的 React 18 单页应用，后端是 `server/` 下的 Fastify 5 ESM 服务。
- 服务启动顺序是：执行数据库迁移 → 写入幂等种子数据 → 装配 Fastify 路由 → 启动问题同步调度器。
- 数据库默认 SQLite，也正式支持 TDSQL/MySQL 兼容库。两个数据库都是发布目标，不能将“本地 SQLite 能跑”当作完成。
- 所有业务 API 位于 `/api` 下，成功响应固定为 `{ code, data, message }`；前端由 `web/src/api/client.js` 自动注入 JWT、CSRF 请求头、解包响应并处理 401。
- 需求和工单共享开发、测试、投产链路：后续任务表的 `req_code` 表示“关联工作项编号”，不只代表需求。
- 投产任务的稳定身份是“对象编号 + 申请投产点”，而不是对象编号本身。
- 状态、字典、角色权限、编号模板、动态必填项均有运行时配置。将它们写死在页面或路由里会制造不可维护的分叉。

## 2. 架构与边界

```text
浏览器
  └─ React 页面 / 编辑器
       ├─ 路由、菜单、Zustand 全局状态
       ├─ 公共 UI：表格、筛选、选择器、附件、状态、编辑器外壳
       └─ API client
             │
             ▼
Fastify（app.js）
  ├─ 安全插件、JWT、RBAC、错误处理、/api 路由注册
  ├─ modules：业务接口和领域编排
  └─ lib：跨模块规则与基础能力
             │
             ▼
db/index.js（唯一业务数据入口）
  ├─ SQLite provider + dialect
  └─ TDSQL provider + dialect
             │
             ├─ migrations / seed
             ├─ 附件、签名文件
             ├─ Excel / Word 导入导出
             └─ PAMS 外部问题同步
```

### 2.1 分层规则

| 层 | 可以做什么 | 不应做什么 |
| --- | --- | --- |
| 页面与编辑器 | 展示、收集输入、调用 API、使用前端权限控制、处理加载/空态/错误态。 | 拼接 SQL、决定最终权限、复制字典解析或附件上传规则。 |
| 路由模块 | 定义 HTTP 入口，校验输入，编排领域服务和持久化，返回统一响应。 | 在多个模块复制跨领域规则或数据库方言判断。 |
| `lib/` 公共服务 | 承载可复用的领域规则、数据转换、文件、审计、导入导出。 | 依赖 React、读取 HTTP 表现层状态。 |
| `db/` | 适配连接、事务、方言与迁移。 | 放页面或业务流程判断。 |
| 迁移与种子 | 只追加架构演进、提供新环境默认值。 | 改写历史迁移、用种子覆盖生产用户配置。 |

## 3. 推荐的职责域划分

职责域是**降低冲突的默认安排**，不是固定编制或审批墙。团队可按人数、任务规模和交付阶段为每个职责域安排一名或多名负责人；一个需求跨域时由发起人负责拉齐接口契约，并在开发前在任务中明确主责和协作者。

| 角色 | 主责范围 | 可并行交付示例 | 需提前同步的热点 |
| --- | --- | --- | --- |
| 交付链路 | 需求、工单、开发、测试、影响/覆盖分析、版本概览。 | 新字段、承接规则、测试阶段能力、概览口径。 | `work-items.js`、状态/必填项、`Overview.jsx`、迁移。 |
| 投产与集成 | PAMS 问题、投产申请、投产审批、会签、签名、附件、Word/Excel 投产材料。 | 新制品属性、会签规则、外部同步、审批导出。 | `release/routes.js`、附件字段键、审批表结构、迁移。 |
| 平台与体验 | 认证、用户/角色/设置、字典/系统/投产点、仪表盘、应用框架、通用组件和主题。 | 权限能力、配置项、图表维度、表格和表单体验、响应式。 | `app.js`、`app.jsx`、`styles.css`、`api/client.js`、迁移。 |

### 3.1 公共热点的串行规则

下列文件或概念发生改动时，只指定一位“集成负责人”落地，其他人的改动以小补丁或后续提交接入，避免三方同时改大文件：

- `server/src/app.js`：新模块注册、安全插件、全局错误处理。
- `server/src/db/index.js`、`providers/`、`dialects/`：数据库能力与兼容性。
- `server/src/db/seed.js`：默认角色、权限、字典与系统配置。
- `server/src/lib/perm-catalog.js`、`required-fields.js`、`status.js`：全局业务规则。
- `web/src/app.jsx`、`router/routes.jsx`、`router/menu.js`：路由、菜单与守卫。
- `web/src/stores/app.js`、`api/client.js`、`main.jsx`、`styles.css`：前端基础设施。
- 两套 migrations：同一个逻辑变更的 SQLite 与 TDSQL 脚本必须一起评审。

共享热点不是禁止修改；它要求先在任务或群组中声明“谁合并、接口如何变化、其他分支基于哪个提交开发”。

## 4. 并行协作流程

### 4.1 开工前的最小设计卡

每个非微小任务在编码前补齐以下六项，放到任务描述或 PR 描述中：

1. **用户结果**：用户完成什么动作，在哪个投产点或阶段生效。
2. **影响范围**：页面、接口、表、公共服务、权限和文档。
3. **数据契约**：新增/变更字段、类型、空值、默认值、兼容旧数据策略。
4. **规则契约**：状态流转、必填项、审计、删除限制、关联对象和投产点口径。
5. **安全契约**：谁能查看、创建、编辑、调整状态、导入导出或签署。
6. **验证路径**：最少一条从 UI 到 API 到数据库的正向路径和一条无权限/非法输入路径。

### 4.2 分支与提交

- 每项工作从最新集成分支拉出短生命周期分支，建议命名 `codex/<owner>/<scope>`，例如 `codex/a/test-coverage`。
- 一个提交只表达一个可理解的变化；不要把格式化全仓、重命名和业务功能混在同一提交。
- 提交消息用“范围: 动词 + 结果”，例如：`release: 校验会签签名归属`。
- 不直接推送或修改他人的进行中分支；发现重叠时先协调由谁吸收冲突。
- 合并前先更新目标分支，解决冲突时优先保留两个功能的完整业务规则，而非只让编译通过。

### 4.3 每日集成节奏

1. 开始：公布今天的主责文件、接口变更和迁移计划。
2. 中段：只要 API、字段、权限键或迁移编号变化，立即通知其他两人。
3. 收尾：PR 中写清测试命令、手工验证、已知限制和是否需要配置变更。
4. 合并：集成负责人按“数据库 → 后端 → 前端 → 文档”的顺序处理跨域变更，合并后跑基础验证。
5. 发现回归：优先回滚可疑的小提交或补一个最小修复；不要在主分支上叠加未经验证的大重构。

## 5. 后端实现准则

### 5.1 API、错误与权限

- 使用 `ok(data, message)` 返回成功数据；使用 `badRequest`、`unauthorized`、`forbidden`、`notFound` 表达预期错误。不要手写不一致的响应结构。
- 任何受保护路由必须使用 `fastify.authenticate`、`fastify.requirePerm(module, action)` 或按测试类型动态调用的权限函数。前端 `<Can>` 只改善体验，不能代替后端保护。
- 新权限先同时更新 `lib/perm-catalog.js`、`db/seed.js` 中模块动作默认值、前端菜单/按钮和路由保护。权限键采用 `模块:动作`，如 `release:release.signoff`。
- 写接口在生产环境会受 CSRF 自定义头校验；所有前端请求继续通过 `api/client.js`，不要另建未带拦截器的客户端。
- 所有用户输入均需服务端校验。SQL 值使用参数绑定；动态列名、排序字段和筛选字段必须经过白名单。

### 5.2 列表、CRUD 与事务

- 通用分页请求口径：`{ page, pageSize, keyword, filters, sort }`。可复用列表优先调用 `listQuery`，其列白名单是 SQL 注入防线的一部分。
- 字典、系统、投产点等简单资源优先用 `registerCrud`；用户、需求、工单、任务、审批等复杂领域继续维护专属路由。
- 关联写入必须使用 `tx()`；不可只依靠前端顺序提交。SQLite 与 TDSQL 的事务由 provider 统一处理。
- 新建、修改、删除的业务实体要接入 `auditCreate`、`auditUpdate`、`auditDelete`。字段标签使用中文，避免日志出现难懂的列名。
- 需要复用需求/工单查询时调用 `getWorkItem` 等 `work-items.js` 能力，不能只查 `requirement` 表而遗漏工单。

### 5.3 状态、必填与流程不变量

- 流程状态来自 `dict_item(process_status)`，终态语义由 `status.js` 缓存和判断。新增状态要同时考虑显示顺序、阶段、初始/进行中/终态分类。
- 更新状态时调用 `assertStatusChangePermission`：只有状态权限的用户不得借状态接口修改普通字段。
- 各模块在不同状态的动态必填规则由 `required-fields.js` 读取 `app_config`。保存前使用 `validateRequiredFields`；附件字段还要校验允许的输入方式。
- 需求或工单一旦被开发/测试承接，编号和删除受保护；不得用新增接口或批量导入绕开该约束。
- NFT、SEC 的不存在是“未按需承接”，不能将空列表解释为异常。
- 投产审批只按“对象编号 + 申请投产点”读写。评审状态、会签结果和投产状态之间存在联动，修改时必须走既有聚合与同步逻辑。

### 5.4 公共服务目录

| 服务 | 责任 | 何时必须使用 |
| --- | --- | --- |
| `db/index.js` | `get/all/run/exec/tx` 与数据库选择。 | 所有业务数据库访问。 |
| `query.js` | 白名单分页、筛选、排序、关键字搜索。 | 标准列表接口。 |
| `crud.js` | 简单资源的 CRUD、权限和留痕。 | 配置型资源。 |
| `audit.js` | 创建/更新/删除的字段级留痕。 | 可追溯的业务写入。 |
| `status.js` / `status-permission.js` | 状态语义与状态编辑权限。 | 所有阶段状态变化。 |
| `required-fields.js` | 动态必填、附件输入模式。 | 所有需配置化校验的保存。 |
| `work-items.js` / `window.js` | 需求+工单统一查询、投产点过滤。 | 跨工作项或窗口统计。 |
| `code-gen.js` | 需求、工单、任务、申请单编号。 | 自动生成编号。 |
| `attachment.js` / `signature.js` | 文件、签名保存、读取与删除。 | 附件、电子签名。 |
| `excel.js` / `io.js` | Excel 解析、导出、通用导入导出。 | 批量数据能力。 |
| `resolver.js` / `dict-cascade.js` | 文本到字典/系统/投产点解析与级联改名。 | 导入、配置改名。 |
| `impact-schema.js` | 影响性分析与测试覆盖的字段模型。 | 开发影响项、测试覆盖。 |
| `chart-dims.js` | 仪表盘数据源、维度、过滤、聚合。 | 新图表指标或维度。 |

## 6. 前端实现准则

### 6.1 路由、权限与全局状态

- 页面先在 `router/routes.jsx` 注册；有导航入口时再更新 `router/menu.js`。受保护区域由 `app.jsx` 根据模块 `view` 权限守卫。
- 使用 `useAppStore` 读取用户、平台配置、主题、投产点和权限。不要在多个页面独立请求并缓存同一份全局用户信息。
- 默认筛选的投产点由全局选择器驱动；页面传递 `releasePointIds` 时保留数组语义，不能偷偷降级成单值。
- 多页签模式会保留打开页面；新组件的副作用必须能在切换、刷新、关闭时正确清理。

### 6.2 公共组件与页面质量

- 列表使用 `DataTable`；复杂筛选使用 `FilterPanel`；配置页优先使用 `CrudManager`。不要复制分页、防竞态或移动端卡片逻辑。
- 表单字段优先选用 `DictSelect`、`SystemSelect`、`PersonPicker`、`AttachmentField`。这些组件封装了现有数据口径和弹层容器行为。
- 状态始终使用 `StatusBadge`，链路进度使用 `ChainBar`；不要手写颜色映射。
- 编辑器以 `EditorShell` 为基本骨架，保留历史记录入口、脏数据提示和统一弹层容器。
- 所有异步交互应覆盖加载、无数据、请求失败、权限不足和保存失败。对删除、清空、驳回等不可逆操作使用二次确认。
- 每个新界面都要检查三种宽度：完整桌面、平板、手机。手机列表提供 `mobileCard`，而不是只保留横向滚动表格。
- 主题色使用 `--radar-*` 变量，状态语义使用状态变量；不要写死“某颜色等于完成”。

### 6.3 前后端契约变更顺序

1. 先确定请求/响应 JSON、权限键、错误文案和向后兼容策略。
2. 后端实现校验、权限、审计和测试。
3. 前端接入 API、显示校验和权限控制。
4. 同时核对列表、详情、编辑、导入、导出与仪表盘/概览是否都需要新字段。
5. 合并前用真实的最小业务数据走一遍 UI → API → 数据库 → 刷新后回读。

## 7. 数据库与迁移规则

### 7.1 迁移是不可变历史

- **绝不修改已发布的迁移文件**，也不通过编辑种子脚本修复已存在生产数据。
- SQLite 迁移在 `server/src/db/migrations/`，TDSQL 迁移在 `server/src/db/migrations/tdsql/`。两侧的编号历史独立：新增逻辑变更使用各自目录的下一个编号，但文件描述应能一一对应。
- 一项 schema 变更必须同时提供两侧 SQL，明确主键、自增、默认值、时间、JSON、索引和字符集差异。
- 迁移在每个文件内运行事务并在 `_migrations` 记录；脚本必须能在空库和已有库上安全执行一次。
- 需要数据回填时，写可重复运行、可识别已处理记录的语句；大批量更新要评估锁表与执行窗口。

### 7.2 双数据库测试点

- SQLite 使用 `node:sqlite` 同步底层 API，TDSQL 使用连接池和 MySQL 协议；业务层仍一律 `await`。
- JSON 读取可能是字符串或对象，使用 `parseJsonArray`、`parseJsonObject` 等工具。
- JSON 查询、日期函数、保留字与占位符有方言差异。业务代码需要差异时先扩展 `dialect` 或 provider，不要加入散落的 `if (DB_CLIENT)`。
- 每个涉及 SQL 或迁移的 PR 至少在 SQLite 跑通；准备发布到 TDSQL 的功能还必须对 TDSQL 完成迁移、启动和关键写入验证。
- 数据库之间迁移数据请执行 [MIGRATION.md](MIGRATION.md) 的流程；数据库迁移永远不包含附件文件。

## 8. 安全、文件与外部集成

- 不提交 `.env`、JWT、数据库口令、PAMS 密钥、真实附件、签名或生产数据。
- 保持 Helmet、CORS、限流、上传大小与扩展名白名单配置；不要为方便测试而在业务代码里绕开它们。
- 附件必须经过 `attachment.js` 保存并绑定实体类型、实体 ID、字段键和上传人；读取/删除同样通过附件模块。禁止把客户端路径直接作为服务器文件路径处理。
- PAMS 只通过 `lib/pams.js` 与 `issues` 模块交互。同步失败应可见、可重试，不能影响本地业务数据的完整性。
- 电子签名必须验证签名归属当前用户；会签必须验证角色与审批对象关系。任何会签规则变化都要补权限和审计验证。

## 9. 测试与验收门槛

当前仓库提供 `server/test/lib.test.js` 的 Node 原生测试；它不足以覆盖所有业务变化。每次提交至少按改动类型完成以下验证。

| 改动类型 | 最低验证 |
| --- | --- |
| 纯文档或样式 | 前端生产构建；明暗主题和一处窄屏人工检查。 |
| 前端页面/组件 | 构建；加载、空态、失败态、无权限态；桌面和手机各一条路径。 |
| 普通 API | 后端单元测试或新增针对性测试；权限拒绝、非法输入、正常写入与回读。 |
| 状态/必填/权限 | 至少验证有权限成功、无权限拒绝、非法状态拒绝、终态必填拒绝。 |
| 数据库/导入导出 | SQLite 迁移后新建和升级库各一次；导入、导出、旧数据读取；TDSQL 发布前验证。 |
| 投产、会签、附件、PAMS | 端到端手工路径，含错误分支与审计日志核对。 |
| 仪表盘 | 不同投产点、空数据、筛选、钻取、个人/系统图表权限。 |

提交前共同基线：

```bash
npm test --prefix server
```

## 11. 何时必须停下来协调

在以下情况不要凭个人判断继续扩张改动范围，应先与相关负责人同步并指定集成负责人：

- 同时变更工作项关联方式、投产点口径或审批任务主键。
- 新增/重命名权限键、流程状态、动态必填项字段或附件字段键。
- 修改 `app.js`、`db/index.js`、provider/dialect、seed、全局 store、路由、主题或全局样式。
- 新增数据库迁移、需要数据回填，或 TDSQL 与 SQLite 的 SQL 无法等价。
- 变更 PAMS 字段映射、会签角色、签名校验、文件存储或导入覆盖策略。
- 一个需求跨越两个以上职责域，且无法用明确 API 契约切开。

协调后的结论必须写入任务或 PR：谁实现、谁集成、谁验证、接口/迁移如何兼容。这样多人并行的速度才不会以流程一致性和可追溯性为代价。
