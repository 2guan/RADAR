# RADAR 界面设计规范

## 设计目标

RADAR 是金融研发交付的工作台，而不是营销页面。界面应呈现“**精确、沉稳、高密度、可核对**”的感觉：用户能快速扫描状态和负责人，在不拥挤的前提下处理大量字段，并随时追溯依据。

产品采用固定侧栏、顶栏和内容工作区的企业控制台布局；PC 是主场景，PAD 与手机保留关键查看和操作能力。实现以 `web/src/main.jsx`、`web/src/theme/presets.js` 和 `web/src/styles.css` 为准。

## 主题与颜色

当前系统提供八套可切换主题：蔚蓝（默认）、滇红、暮橘、松翠、澄碧、楹紫、山茶和苍青，并且每套都提供明暗两种模式。**不要假设品牌主色固定为翡翠绿，也不要在组件中硬编码某个主题色。**

新增样式应使用主题变量：

| 用途 | CSS 变量 |
| --- | --- |
| 主操作、激活、链接 | `--radar-primary`、`--radar-primary-deep` |
| 主色弱化背景和强调 | `--radar-primary-soft`、`--radar-primary-fade` |
| 初始、进行中、终态 | `--radar-status-initial`、`--radar-status-in-progress`、`--radar-status-final` |
| 页面、表面、文本、边框 | `--radar-bg`、`--radar-surface`、`--radar-ink`、`--radar-text-secondary`、`--radar-border` |
| 补充强调 | `--radar-highlight`、`--radar-accent` 及对应 `soft` / `fade` 变量 |

状态表达必须由 `StatusBadge` 和中文文字承载，颜色只用于加强，不得成为唯一信息来源。图表优先复用 `components/dashboard/chartOption.js` 和 `ColorPickerField.jsx` 的调色规则。

## 版式与层级

- **应用框架**：侧栏承载按权限过滤的导航；顶栏提供页面上下文、投产点和用户操作；内容区支持普通模式和多页签模式。
- **信息密度**：默认正文为 14px；表格使用紧凑行高和横向滚动，窄屏转换为卡片。先压缩冗余操作，再减少信息，不用大面积留白掩盖复杂度。
- **间距**：沿用 4 / 8 / 12 / 16 / 20 / 24px 的现有节奏；同一区块内的标题、筛选、表格和操作区对齐。
- **容器**：现有主题采用近直角的专业感，Ant Design 的大容器圆角为 0，小控件约为 2px。新增卡片、弹窗、页签和按钮不要任意改成大圆角或胶囊形。
- **层级**：以实色表面、细边框和轻阴影划分层次；不要加入玻璃拟态、强渐变背景、大面积投影或无意义动效。

## 组件使用规则

| 场景 | 先用什么 | 约束 |
| --- | --- | --- |
| 列表 | `DataTable` | 统一关键字搜索、排序、分页、刷新、列宽拖拽和移动端卡片；`fetcher` 返回 `{ list, total }`。 |
| 配置 CRUD | `CrudManager` | 用于字典、系统、投产点、角色等简单资源，不将复杂业务流程硬塞进去。 |
| 筛选 | `FilterPanel` | 筛选项以业务口径组织，投产点支持多选时须传递完整数组。 |
| 状态 | `StatusBadge`、`ChainBar` | 状态中文文案、流程节点和阶段颜色必须保持一致。 |
| 表单选择 | `DictSelect`、`SystemSelect`、`PersonPicker`、`SystemNameInput` | 不在业务页重复实现字典、系统或人员查询。 |
| 详情与编辑 | `EditorShell`、`editors/` | 保持原位处理与历史入口；删除/驳回使用明确的二次确认。 |
| 附件与签名 | `AttachmentField`、`SignaturePad` | 使用现有实体、字段键和权限规则，不能直接操作文件路径。 |
| 权限 | `Can` | 用于按钮和区域可见性；后端权限仍必须同步实现。 |

## 页面模式

1. **列表页**：筛选与操作在表格上方；状态、编号、负责人、系统和投产点优先可扫描；行点击进入详情或编辑器时保持语义明确。
2. **编辑器**：按业务阶段分组字段，必填标记、校验错误和附件要求紧邻字段；脏数据取消需给出可理解的提示。
3. **详情页**：先展示对象身份和当前状态，再展示阶段信息、关联对象、附件和审计历史；避免将关键状态藏在折叠区底部。
4. **仪表盘**：图表先讲清统计口径、投产点范围与过滤条件，再呈现结果；钻取明细必须能返回对应业务记录。

## 响应式与可访问性

- `>=1024px` 使用完整工作台；`768–1023px` 收紧间距并保持主要操作；`<768px` 侧栏转抽屉，表格优先使用 `mobileCard`。
- 交互元素支持键盘焦点和清晰焦点态；图标按钮必须提供文字或 Tooltip。
- 明暗主题都要检查正文、次级文字、边框、状态徽章和图表的可读性，目标至少满足 WCAG AA 的常规对比度要求。
- 界面文案统一中文，日期、状态和投产点使用已有格式化工具，避免同一概念多种写法。

## 禁止事项

- 不要直接写固定蓝、绿、灰作为主题或状态色；不要用颜色代替状态文字。
- 不要为单页复制一套表格、选择器、附件或权限实现。
- 不要把桌面表格简单缩小到手机；应提供已有的卡片表现或明确的移动端信息优先级。
- 不要将视觉偏好覆盖业务语义：投产点、状态、审计、会签和附件应始终容易找到。
