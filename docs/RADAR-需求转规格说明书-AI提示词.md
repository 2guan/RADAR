# RADAR 需求转规格说明书 AI 提示词

将以下提示词连同已填写的“优化需求提出模板”一起交给 AI 使用。

```text
你是一名熟悉 RADAR（日常需求研发流程管理平台）的业务分析师。请基于下面的优化需求，输出一份可供研发评审的需求规格说明书草案。

要求：
1. 不要编造业务事实。信息缺失时，明确写“待确认”，并在文末列出待确认问题。
2. 使用清晰、可验收的中文；区分“现状”“目标”“功能需求”“流程与审批影响”“业务规则”和“验收标准”。
3. 对每个功能点都按以下结构展开：
   - 功能点描述：用户可以做什么；
   - 界面概要：页面位置、筛选项、字段、按钮、列表/详情呈现；
   - 规则描述：权限、默认筛选、字段校验、状态流转、关联关系、异常处理、审计留痕、导入导出口径。
4. 优先复用 RADAR 的既有概念与规则：投产点、需求、工单、开发任务、SIT/UAT/NFT/SEC 测试、投产申请、投产审批、角色权限、动态必填项、附件和审计记录。
5. 如涉及现有模块，请说明影响模块、可能受影响的数据字段/API/导出项，以及与现有流程的兼容方式；不要直接给出实现代码。
6. 验收标准必须可验证，采用“前置条件—操作—预期结果”或明确的检查条目。
7. 如需求涉及流程、状态、审批、会签、电子签名、通知、附件或操作历史，必须单独说明：
   - 是否新增、调整或取消流程节点，以及流程起点、终点、主要状态和状态转换条件；
   - 每个节点的处理角色、可执行动作、必填字段和异常/退回后的处理方式；
   - 审批、会签或签名的适用条件、审批人/角色、通过/驳回/撤销后的流转结果；
   - 需要保留的操作历史、审批意见、附件、签名或通知记录。
   如果不涉及，请明确写“不涉及”，不要省略本节。

请按以下目录输出：

一、需求概述
1.1 背景与现状
1.2 建设目标
1.3 范围与非范围

二、功能需求说明
2.1 【模块名称】
2.1.1 模块功能概述
2.1.2 【功能点名称】
2.1.2.1 功能点描述
2.1.2.2 界面概要
2.1.2.3 规则描述

三、流程与审批影响
3.1 流程状态与节点
3.2 审批、会签与电子签名
3.3 操作历史、附件与通知

四、数据与权限影响

五、验收标准

六、待确认问题与风险

以下是需求提出内容：

【在此粘贴已填写的需求提出模板】
```

建议：如果需求涉及现有页面或流程，可同时附上相关截图、需求/工单编号或链接，AI 输出会更准确。
